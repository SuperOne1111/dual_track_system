## 四级行业双轨筛选系统 - 项目实施计划

## 1. 项目概述

| 项目属性     | 说明                                                         |
|:-------- |:---------------------------------------------------------- |
| **项目名称** | 四级行业双轨筛选系统 (Four-Level Sector Dual-Track Screening System) |
| **核心目标** | 构建分散但不过度分散的行业轮动组合，实现事件驱动调仓与超额收益                            |
| **关键约束** | **零 DDL** (仅 3 张只读表)、**双轨物理隔离**、**脚本化运行**、**防未来函数**        |
| **运行模式** | Backtest (内存状态) / Production Daily (文件状态)                  |
| **参考文档** | 《数据库结构说明书》、《接口架构设计》、《技术规格说明书》  |

## 2. 项目阶段划分与里程碑

| 阶段     | 名称           | 周期    | 核心目标                       | 关键交付物                                | 合规门禁                     |
|:------ |:------------ |:----- |:-------------------------- |:------------------------------------ |:------------------------ |
| **P1** | **环境与数据准备**  | T+1 周 | 验证数据库只读连接，确认字段映射           | `config.yaml`, `utils/validation.py` | **DB-01/02**: 无写权限，无 DDL |
| **P2** | **核心模块开发**   | T+3 周 | 完成 7 层架构代码实现，双轨隔离          | `modules/` 下 7 个 py 文件               | **STRAT-01**: 趋势/左侧类独立   |
| **P3** | **状态与一致性构建** | T+4 周 | 实现 Repository 模式，确保回测/实盘一致 | `modules/state_layer.py`, `tests/`   | **CONS-02**: 核心逻辑无分支     |
| **P4** | **回测与验证**    | T+5 周 | 完成历史回测，验证因子 IC 与风控         | `tests/test_end_to_end.py`, 回测报告     | **CONS-01**: 无未来函数       |
| **P5** | **实盘部署与监控**  | T+6 周 | 脚本化部署，状态文件原子写入验证           | `main.py`, `data/*.json`             | **RUN-01/02**: 脚本化，原子写入  |

## 3. 任务分解结构 (WBS) - 基于文件架构

本 WBS 严格对应《接口架构设计》第 6 节目录结构。

### 3.1 阶段 P1: 环境与数据准备

| 任务 ID     | 任务名称    | 对应文件/模块               | 详细说明                                               | 约束检查                            |
|:--------- |:------- |:--------------------- |:-------------------------------------------------- |:------------------------------- |
| **P1-01** | 数据库连接配置 | `config.yaml`         | 配置 PostgreSQL 只读账号，设置 `pit_enabled: true`          | 确认账号无 CREATE/ALTER 权限           |
| **P1-02** | 字段映射验证  | `utils/validation.py` | 编写 SQL 校验脚本，确认 3 张表字段名与文档完全一致                      | **DB-01**: 禁止 `SELECT *`，必须显式字段 |
| **P1-03** | 数据质量探查  | `utils/validation.py` | 检查 `fin.daily_sector_industry_cn` 缺失率 (<5%) 与主键唯一性 | 符合《数据库结构说明书》4.1 节               |

### 3.2 阶段 P2: 核心模块开发 (7 层架构)

| 任务 ID     | 任务名称  | 对应文件/模块                      | 详细说明                                                                   | 约束检查                          |
|:--------- |:----- |:---------------------------- |:---------------------------------------------------------------------- |:----------------------------- |
| **P2-01** | 数据层实现 | `modules/data_layer.py`      | 实现 `DataLoader` (含 `as_of_date`) 与 `SectorMapper`                      | **DB-03**: 所有接口含 `as_of_date` |
| **P2-02** | 特征层实现 | `modules/feature_layer.py`   | 实现 `FeatureEngine`，区分 `compute_trend_factors` 与 `compute_left_factors` | **STRAT-01**: 轨道内独立标准化        |
| **P2-03** | 筛选层实现 | `modules/screening_layer.py` | 实现 `TrendScreener` (漏斗) 与 `LeftScreener` (候选池)                         | **STRAT-01**: 物理隔离，无共用筛选器     |
| **P2-04** | 组合层实现 | `modules/portfolio_layer.py` | 实现 `PortfolioBuilder.merge_tracks`，硬编码 0.85/0.15                       | **STRAT-02**: 权重比例固化          |
| **P2-05** | 风控层实现 | `modules/risk_layer.py`      | 实现 `RiskManager` (止损) 与 `CoolDownManager` (10 日冷却)                     | **MECH-01**: 冷却池持久化逻辑         |
| **P2-06** | 执行层实现 | `modules/execution_layer.py` | 实现 `OrderGenerator` 与 `BacktestEngine`                                 | **RUN-01**: 无常驻服务逻辑           |

### 3.3 阶段 P3: 状态与一致性构建 (关键合规点)

| 任务 ID     | 任务名称   | 对应文件/模块                  | 详细说明                                                     | 约束检查                            |
|:--------- |:------ |:------------------------ |:-------------------------------------------------------- |:------------------------------- |
| **P3-01** | 状态抽象层  | `modules/state_layer.py` | 定义 `StateRepository` 接口，实现 `Memory` 与 `File` 子类          | **CONS-02**: 核心逻辑无 `if mode` 分支 |

## 4. 关键合规性检查点 (Compliance Gates)

在代码合并或进入下一阶段前，必须通过以下检查（基于 13 项强制标准）：

| 检查点编号        | 检查阶段  | 检查内容                                            | 否决标准                             |
|:------------ |:----- |:----------------------------------------------- |:-------------------------------- |
| **CG-DB**    | P1 结束 | 确认代码中无 `CREATE`, `ALTER`, `INSERT`, `UPDATE` 语句 | 出现任何写库语句 → **不通过**               |
| **CG-STRAT** | P2 结束 | 确认 `TrendScreener` 与 `LeftScreener` 无共用内部方法     | 发现共用筛选逻辑 → **不通过**               |
| **CG-CONS**  | P3 结束 | 审查 `main.py` 及核心流程，确认无 `if mode == backtest` 分支 | 核心逻辑存在模式分支 → **不通过**             |
| **CG-RUN**   | P5 开始 | 确认 `FileRepository.save` 使用原子写入流程               | 直接 `open(..., 'w')` 覆盖 → **不通过** |

## 5. 风险管理计划

| 风险项               | 可能性 | 影响程度 | 应对措施                                             |
|:----------------- |:--- |:---- |:------------------------------------------------ |
| **数据未来函数**        | 高   | 致命   | 严格执行 `as_of_date` 切片，P4 阶段进行随机时点穿透测试             |
| **状态文件损坏**        | 中   | 高    | 强制使用 `utils/io.py` 原子写入，增加校验和验证                  |
| **双轨逻辑污染**        | 中   | 高    | 代码 Review 时重点检查 `screening_layer.py`  imports 关系 |
| **数据库 schema 变更** | 低   | 高    | 锁定数据库账号权限为只读，应用层禁止 DDL                           |
| **因子 IC 失效**      | 中   | 中    | 启用 `deployment.ic_monitoring` 机制，自动降权            |
